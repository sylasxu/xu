# xu 产品需求文档

> **版本**：正式版（State Home + Agent Core + GenUI Runtime）
> **更新日期**：2026-04-18
> **App 名称**：xu
> **产品定位**：群主分身型组局产品（对外） / 持续任务运行时（对内实现）
> **核心 Slogan**：今天有什么想玩的？
> **AI 角色名**：xu

---

## 0. 产品哲学

### 0.1 产品定义

xu 的产品定义围绕三个核心问题展开：

1. **产品本质是什么**
2. **首页承担什么职责**
3. **生成式 UI 管到哪一层**

#### 结论

xu 不应该是：

- 传统活动平台 UI 上硬加一个 AI
- 也不应该是只有一个空白聊天框的 ChatGPT 壳

而应该是：

## **一个以 agent 为核心、以状态首页为入口、以生成式 UI 完成任务的组局产品**

再说白一点：

**首页负责判断“现在最需要被接住的事”**  
**agent 负责理解需求、调用数据、推进流程**  
**生成式 UI 负责把结果、进度、下一步动作自然地长出来**

这构成 xu 的完整产品结构。

### 0.4 第一性用户场景

- “有没有 xx 搭子”
- “附近有没有局”
- “我想组个局”
- “麻将差一个有没有人”
- “这场结束了，要不要复盘 / 再约”

xu 不是从抽象 AI 往场景里找落点，而是从这些高频、口语化、结果导向很强的真实表达出发，设计一个能持续接住它们的碎片化社交助理。

### 0.5 唯一大脑原则

- 用户看到的是一个“xu”，系统内部也必须尽量表现为一个持续拥有任务的运行时，而不是若干页面和按钮在接力
- 对同一件事，系统需要有一个持续存在的任务对象，至少记录：`用户目标`、`当前阶段`、`下一步动作`、`最后结果`
- `join_activity`、`find_partner`、`create_activity` 都必须进入同一套任务运行时体系
- 登录 / 绑手机号不应被视为任务终止，而应视为任务挂起点；恢复后必须继续推进同一条任务

## 1. 产品本质与定位

### 1.1 首页职责定位

首页设计不是“活动首页”和“纯 chat”之间的二选一，而是对首页职责的重新定义：

### **首页不是活动广场**

因为产品早期供给不稳定，而且产品也不是靠“让用户刷活动”取胜。

### **首页也不是纯 chat**

因为用户第一次进入时，或者回到产品时，往往不是要从零组织一句话，而是要被续接。

### **首页应该是一个状态感知的 agent 入口**

也就是：

- 空的时候，帮用户快速开始第一次提问
- 有事情进行中时，帮用户继续上次的任务
- 有活动快开始时，直接把重要事项顶出来
- 有活动刚结束时，承接评价或再组一局

因此，首页的核心问题不是“展示什么内容”，而是：

## **首页作为状态容器，如何承接 agent 的上下文**

### 1.2 产品本质

xu 不是一个泛泛的“同城活动平台”，  
也不是一个泛 AI 社交助手。

xu 的本质是：

## **把微信群主的工作流产品化**

具体是把这几件事数字化：

- 找现成的局
- 发起一个局
- 差人补位
- 帮用户筛选和匹配
- 替代大量重复私聊
- 帮不敢在群里说话的人开口
- 沉淀用户信用和质量

因此，产品定义是：

## **群主分身 + 组局操作系统**

这比“首页展示活动”或者“像豆包一样聊天”都更准确。

### 1.3 一句话定位

**一个把“找搭子、找活动、补人、发局”数字化的 agent 产品。**

### 1.4 核心价值

对用户：

- 不用翻群
- 不用尬聊
- 不用私聊群主反复确认
- 直接说一句话就能办事

对组织者：

- 自动处理重复问答
- 自动补位
- 自动匹配
- 自动承接需求
- 自动沉淀状态和信用

### 1.5 三大核心优势

#### 响应感

| 旧世界（微信群） | 新世界（xu） |
|-----------------|---------------|
| 发消息后死一般的寂静 | AI 秒回，0.5s 内开始响应 |
| 运气好有人回表情包 | 流式反馈：文字逐字显示，Widget 卡片逐步展开 |

#### 秩序感

| 旧世界（微信群） | 新世界（xu） |
|-----------------|---------------|
| 信息是流（Flow），瞬间被冲走 | 信息是卡片（Widget），24 小时内雷打不动 |
| 想找刚才那个人？翻聊天记录 | 对话历史持久化，下次打开还在 |

#### 零门槛

| 旧世界（填表单） | 新世界（对话输入） |
|-----------------|-------------------|
| 选时间、选地点、填标题 | 在群里怎么说话，在这里就怎么说话 |
| 需要学习新操作 | 直接把群里的乱七八糟复制过来，自动变整齐 |

### 1.6 差异化定位

这章不再做长篇竞品攻防，而只保留对产品精神内核有帮助的判断。

#### 不是在做什么

xu 不是：

- 一个泛泛的同城活动平台
- 一个靠 feed 和货架成立的活动展示产品
- 一个只有空白聊天框的通用 AI 助手

#### 本质模型的差异：商城 vs 秘书

| 维度 | 传统活动平台 | xu |
|------|-------------|----|
| 核心定义 | 活动货架 / 活动电商平台 | 群主分身 / 组局操作系统 |
| 服务对象 | 商家、俱乐部、职业主理人 | 普通群主、朋友、热心群友、半熟人关系网 |
| 为了什么 | 卖票、流量、分发 | 省事、怕冷场、快速成局、接住需求 |
| 操作逻辑 | 用户适应表单和平台结构 | 系统适应用户语言和当前任务 |
| 首页逻辑 | 展示内容和货架 | 续接状态和当前最该处理的事 |

#### 我们要做什么

## **不做平台，做工具**

更准确地说：

- 不做靠内容分发成立的活动广场
- 不做重运营、重俱乐部、重票务的平台结构
- 不做把用户强行拉进另一套复杂后台的系统

而是极致地做：

- 微信群主工作流的数字化
- 对碎片化社交需求的即时承接
- 对普通用户和不敢开口用户的低门槛表达支持
- 对“找局 / 发局 / 补人 / 跟进结果”这套真实工作流的持续推进

#### 场景判断

对 xu 来说，最值得守住的不是“严肃商业活动”，而是那些传统平台做得很重、很慢、很没人情味的场景：

- 朋友聚餐
- 临时起意
- 差一人补位
- 半熟人拼车 / 拼单 / 组局
- 不知道去哪但想先开口问一句

这些场景的共同特点不是“活动规模大”，而是：

- 高频
- 临时
- 低客单价
- 强语境
- 更依赖有人接话，而不是有人展示内容

#### 一句话差异化

如果必须用一句话说清 xu 和典型活动平台的区别：

> **典型活动平台是在卖活动；xu 是在替人把局张罗起来。**

再说白一点：

> **别人做的是活动货架，xu 做的是会接话、会办事、会续上的群主分身。**

## 2. 产品阶段与首页成立条件

### 2.1 冷启动与稀疏供给阶段

xu 首先要在冷启动和稀疏供给条件下成立。

这意味着：

- 首页不能假装自己是一个已经供给稳定、活动密度很高的活动广场
- 首页不能靠“这里现在就有人”这种成熟平台叙事来成立
- 首页必须真诚表达当前真实成立的东西：承接能力、状态续接、任务推进能力
- 供给成熟后的表达升级，是后续阶段能力，不是首页成立前提

### 2.2 首页成立原则

- 首页首先是状态首页，不是活动广场
- 首页可以轻量表达真实动态，但不能依赖伪热闹数据成立
- 首页的核心任务是“接住人”，不是“展示热闹”
- 当供给成熟后，首页可以在不破坏状态主轴的前提下逐步增加真实供给证明层

### 2.3 产品任务

- 不需要推翻产品原点或交互骨架
- 主矛盾不再是“做什么”，而是“把最重要的真实场景做顺、做稳、做得像人”
- 后续判断标准不是“有没有再加功能”，而是“xu 是否更像一个真的会张罗组局的群主”

### 2.4 优先级

1. `找搭子`
2. 首页 / 消息中心的连续承接感
3. 活动后续上的产品感
4. 组局链路的顺手感与情绪价值

## 3. 核心产品结构

### 3.1 产品架构

### 不再是传统 tab 产品思路

你最早的“地图 / 消息 / 我的”，更像平台期架构。  
现在不适合继续拿它做主轴。

### 现在的主轴应该是

## **首页状态容器 + agent 流 + 任务/详情页**

也就是：

### A. 首页

负责：

- 首次进入时快速开始
- 回来时续接任务
- 根据状态展示当前最重要的事

### B. agent 流

负责：

- 理解用户意图
- 调数据库 / 搜索 / 匹配
- 生成 UI
- 推进流程

### C. 详情与任务页

负责：

- 活动详情
- 发起组局
- 查看匹配进度
- 查看报名 / 补位状态
- 管理自己的局

也就是说：

**首页不再是信息广场**  
**首页也不是完整流程**  
**首页是 agent 的入口和任务恢复器**

### 3.2 生成式 UI 与产品结构的分工

- 首页负责承接与续接，不负责完成所有业务动作
- agent 负责理解与执行，不负责决定整页产品结构
- 生成式 UI 负责把结果、进度、下一步动作长出来，不负责生成首页骨架
- 详情页、讨论区、消息中心是任务推进中的专门承接面，不是和 agent 并列竞争的第二套产品

## 4. 首页

### 4.1 首页方案

### 首页固定骨架

- 顶部一句轻动态文案
- 中间一个主状态区域
- 底部常驻输入条

### 首页有两种大模式

#### 模式 A：空态 / 首次进入

中间展示几句真实问题，直接可点击：

- 今晚有没有麻将局？
- 差一个人吃火锅，有人吗？
- 周末想找轻松一点的活动
- 帮我发一个明晚 KTV 局

底部输入条 placeholder：

**告诉我你现在想做什么…**

#### 模式 B：有任务状态

中间不再展示欢迎问句，而是展示主状态卡，例如：

- 正在帮你找人
- 你今晚有个麻将局
- 继续上次的组局
- 这次活动结束啦，想不想再组一个

底部输入条继续兜底。

### 这个首页的本质

## **不是欢迎页，而是状态首页**

### 4.2 首页职责

- 首页负责判断“你现在最需要被接住的事”
- 首页首先处理“该从哪件事开始”，而不是“该先展示多少内容”
- 首页是 agent 的入口和任务恢复器，不是内容广场，也不是孤立欢迎页
- 首页只需要把用户带入同一条主任务链，不需要在首页完成整套流程

### 4.3 首页表达状态机

首页通过有限状态收敛复杂业务情况。

#### 首页表达状态收敛成 5 个

- H0 空态
- H1 待继续
- H2 进行中
- H3 待出发 / 待处理
- H4 收尾态

#### 状态优先级

- H3 待出发最高
- H2 进行中次之
- H1 待继续
- H4 收尾
- H0 空态兜底

#### 原则

- 首页只显示 **1 个主状态卡**
- 允许有一行极轻的次要事项提示
- 底部输入条始终存在
- 当前实现中，H5 首页已经直接消费任务快照真相源，并由服务端返回的 `homeState + primaryTaskId` 决定主状态卡，而不是前端各自猜测首页状态

这套规则的作用，是把复杂回访状态统一收敛为少数几种首页表达态。

### 4.4 首页状态说明

#### H0 空态

- 用户没有明显进行中的任务，或当前没有可优先续接的事项
- 首页重点是帮助用户理解“这里能替我办什么事”
- 表达方式以真实问题示例、轻动态文案、底部输入条为主

#### H1 待继续

- 用户有一件被中断但仍值得继续的任务
- 首页主卡明确指出“继续上次的组局 / 继续刚才的找人”
- 点击后直接回到同一条任务链，而不是重新开始

#### H2 进行中

- 用户已有明确进行中的任务，例如“正在帮你找人”
- 首页主卡提供当前进度、下一步建议、查看进度入口

#### H3 待出发 / 待处理

- 用户有时间敏感或结果敏感的事项
- 例如即将开始的活动、待确认的匹配、待处理的招人进展
- 这是首页最高优先级状态

#### H4 收尾态

- 任务主体已经完成，但需要收尾或进入后续承接
- 例如活动结束后的反馈、复盘、再约

### 4.5 首页与欢迎态

- Welcome 不是单独产品，而是首页在 H0 空态下的一种表达
- Conversation 不是另一个页面，而是首页进入任务主链后的主对话舞台
- 当前首页不再把欢迎内容塞成 `Widget_Dashboard` 消息；欢迎态是页面级结构，不是消息级结构
- 当前首页不再默认展示统计卡、热闹数据、伪供给证明

### 4.6 首页与供给成熟度

- 冷启动和稀疏供给阶段：首页以状态承接为主
- 有真实但稀疏供给阶段：首页可轻量增加真实动态，但不能反客为主
- 供给稳定阶段：首页可以在状态骨架不变的前提下增加真实供给证明层
- 无论哪个阶段，首页都不应退化成传统 feed 广场

### 4.7 首页输入与快速入口

- 快速入口的职责是帮用户起第一句，不是充当功能菜单
- 快速入口优先使用真实问题和可执行表达，而不是抽象功能名
- 输入框是首页唯一持续存在的主 CTA
- 输入框设计应弱化“我要发布”，强化“我可以先开口”

## 5. 生成式 UI 的边界

### 5.1 不应该生成的

- 首页整体布局
- 状态优先级
- 组件种类
- CTA 数量
- 信息层级

### 5.2 应该生成的

- 欢迎语
- 空态示例问题
- 状态摘要
- 次要事项提示
- 结果卡的内容
- 匹配进度描述
- 下一步建议动作

所以最终不是“整页让 AI 自己长”，而是：

## **规则驱动的页面骨架 + 生成式填充内容**

这构成当前最稳定的产品解法。

### 5.3 响应节奏原则

- xu 必须先“说清楚”，再“给动作”；正文是主战场，生成式 UI 是正文之后的承接
- 当一轮回复同时包含说明和交互时，用户应先看到流式文字，再在下一行看到按钮、结果卡、表单或下一步操作
- 生成式 UI 不应重复解释正文已经说过的意思；它只负责承载选择、填写、浏览和执行
- 如果没有结构化交互必要，就只返回正文；不要为了“有 UI”强行补一层卡片

### 5.4 页面骨架与组件层

- 首页骨架、状态切换、页面层级由产品规则决定
- Widget / Result Card / Form / Explore / Action 等 GenUI 组件由任务上下文决定
- 组件是完成任务的承接层，不是首页结构本身

### 5.5 GenUI 组件层

GenUI 组件层应围绕统一的 `GenUI blocks` 协议定义，而不是围绕旧的 `Widget_*` 命名做主叙事。

#### 当前主链稳定块类型

结合当前代码与协议，主链稳定使用的结构化块主要包括：

| Block 类型 | 作用 | 主流程中的典型场景 |
|-----------|------|------------------|
| `text` | 正文说明 | 先说清楚当前理解、解释为什么要补信息、总结结果 |
| `choice` | 轻量选择 | 让用户快速确认方向、选择下一步、补 1 个关键分叉 |
| `entity-card` | 单对象结果卡 | 展示一个活动、一个摘要结果、一个待确认对象 |
| `list` | 多结果列表 / 卡组 | 探索附近活动、搭子搜索结果、可浏览候选集合 |
| `form` | 结构化收集 | 找搭子偏好、发局草稿补充、一次收集多项信息 |
| `cta-group` | 下一步动作组 | 报名、继续、换一批、我也组一个、查看详情 |
| `alert` | 风险 / 错误 / 成功提示 | auth gate、错误提示、成功确认、提醒类信息 |

#### 主流程最需要的块

对当前主链来说，最重要的不是“块种类越多越好”，而是下面这套能覆盖主流程：

- `text`：先把话说清楚
- `list`：给用户看结果
- `form`：一次收集必要信息
- `cta-group`：给用户明确下一步
- `entity-card`：承接单个活动或结果摘要

也就是说，当前主流程的产品重心不是保留尽可能多的旧 Widget，而是让以下能力稳定成立：

- 说清楚
- 给结果
- 补信息
- 推下一步

#### 旧 Widget 命名的处理方式

旧 PRD 和小程序代码中仍然存在一批 `Widget_*` 命名，例如：

- `Widget_Draft`
- `Widget_Explore`
- `Widget_Share`
- `Widget_AskPreference`
- `Widget_Action`
- `Widget_Error`

这些命名可以继续作为实现层和兼容层存在，但 PRD 不再以它们作为产品主叙事。

更准确的关系是：

- `Widget_*` 是历史实现与多端兼容层
- `GenUI blocks` 是当前统一主链协议层

#### ActionResult 的定位

- 成功操作后仍然需要清晰的结构化结果反馈
- 这类结果可以通过 `entity-card`、`alert`、`cta-group` 的组合来表达
- 核心不是“再发一个独立结果系统”，而是让成功反馈和下一步动作在同一条链路里连起来

### 5.6 意图识别与动作路由

意图识别不再采用“P0 热词主导回复路径”的结构。

#### 识别结构

- `P0`：关键词匹配，作为**附加业务信号**
- `P1`：Feature Combination 规则引擎
- `P2`：LLM Few-shot 分类

#### 主策略

## **P1 / P2 意图识别是主策略，P0 热词只作为补充信号。**

也就是说：

- `P0` 命中后不再直接决定整条回复路径
- `P0` 主要用于补充 metadata、观测、统计和部分后置策略
- 真正决定当前输入该进入哪条主链的，仍然是 `P1 / P2` 的意图识别结果

#### 设计原因

因为当前产品定位已经从“直播热词互动”回到“聚焦主流程”：

- 不是先想办法做热词命中
- 而是先把 `找局 / 发局 / 找搭子 / 活动后续上` 这些真实表达接住
- 所以热词能力可以存在，但不应再成为当前 PRD 的主叙事

#### 意图识别原则

- 优先理解用户这句话想完成什么事
- 低置信度时允许从规则引擎升级到 LLM 分类
- 不靠热词系统来替代主链理解
- `unknown` 和兜底场景必须被显式处理，而不是偷偷回退成旧意图

#### 核心意图类型

结合当前主链，意图识别最重要的不是把所有话都分得很花，而是稳定识别这些方向：

- `create`：我想自己发一个局
- `explore`：我想看看现在有什么局
- `partner`：我想找搭子 / 找同行 / 补位
- `manage/query`：我想继续、查看、管理我已有的事
- `follow-up`：我想围绕已经发生的活动继续处理

#### 热词 / Hot Chips 的定位

- `Hot Chips` 和 `P0 热词` 可以继续作为局部快捷入口或后续运营层存在
- 但它们不再是首页主结构，也不再是当前产品成立的前提
- 如果未来自媒体、直播或运营场景重新需要这层能力，可以作为附属层启用

也就是说：

- 不围绕它设计主流程
- 先把主流程写稳

#### 意图识别之后该发生什么

意图识别不是为了分类而分类，而是为了把用户送进正确的下一步：

- `create` → 草稿 / 表单 / 确认发布主链
- `explore` → 列表 / 活动详情 / 报名主链
- `partner` → 搜索结果 / 继续帮我留意 / 匹配详情主链
- `manage/query/follow-up` → 恢复已有任务、详情页、消息中心或活动后链路

## 6. 核心业务流程

### 6.1 结果导向总漏斗

> **北极星不是“聊得久”，而是“完成得快”。**

用户从进入首页开始，必须在一次连续交互里完成至少一个结果：

1. 找到局（报名成功 / 发布成功）
2. 找到搭子（匹配确认 / 进入匹配中状态）

### 6.2 四大真实场景

#### 找局

- 首页 Chat 可进入 explore
- `join_activity` 已形成从探索、报名、auth gate、讨论区到活动后 follow-up 的连续任务
- 报名被打断后可恢复同一条动作
- 报名成功后统一进入讨论区

#### 组局

- `create_activity -> draft_ready -> published -> done` 已形成连续任务
- 草稿真实写库
- `edit_draft / save_draft_settings` 不再分叉新任务
- 发布成功后有下一步 CTA

#### 找搭子

- 当前主链是 `Search-First + Optional Pool`
- 首轮不再默认弹完整 form 或直接入池
- 没有真实 `intent_match` 前不会误进 `match_ready`
- 真匹配后可从消息中心直接承接

#### 活动后续上

- `confirm-fulfillment`、review、rebook、memory 写回已成真链路
- 带 `activityId + followUpMode` 的入口会挂回原活动
- `join_activity` 可推进到 `post_activity -> done`

### 6.3 主流程成熟度排序

1. 找局
2. 组局
3. 活动后续上
4. 找搭子

### 6.4 优先收口点

- `find_partner` 仍未达到“像群主本人一样会判断”的程度
- 首页和消息中心的连续承接感还不够强
- 活动后续上虽已形成闭环，但还没长成特别强的招牌体验

### 6.5 活动全生命周期

活动不是“发布成功”就结束，而必须形成从报名到活动后续上的完整闭环。

#### 当前问题

| 阶段 | 旧状态 | 当前补全 |
|------|--------|----------|
| 创建活动 | 已支持 AI 一句话创建 | 保持不变 |
| 分享活动 | 已支持分享链接/卡片 | 与活动详情合并理解，不再单独依赖“邀请函”心智 |
| 报名 | 已支持一键报名 | 保持不变 |
| 报名后 | 容易出现空白感 | 报名成功后统一进入讨论区 |
| 活动前 | 提醒弱 | 补充活动前提醒 |
| 活动中 | 主要靠讨论区沟通 | 保持不变 |
| 活动后 | 容易结束即空白 | 进入反馈、复盘、再约与真实结果写回 |

#### 报名成功后的统一链路

用户从活动详情、半屏详情、AI 推荐卡等任一入口点击报名后：

- 如未满足动作闸门，先拉起登录 / 绑定手机号
- 报名成功后统一进入讨论区
- 讨论区首屏不能是空白，要有系统消息、轻量引导和默认破冰话术
- `join_activity` 任务必须从 `joined` 推进到 `discussion`

#### 活动前提醒

- 活动开始前，应通过消息中心和可用通知渠道提醒参与者
- 提醒内容必须明确活动标题、时间、地点和当前建议动作
- 提醒应被视为同一条任务的阶段推进，而不是孤立通知

#### 活动后流程

- 活动结束后，系统应承接“复盘 / 再约 / 履约确认”
- 发起人在活动确认页提交真实履约结果（谁到场、谁未到场）
- 系统将真实活动结果写入长期记忆
- 后续 Explore / AI 推荐 / 再约建议，应优先参考真实结果，而不是只参考聊天内容

#### 信号强弱原则

- 报名（join）只算轻信号，不能直接等价于强正反馈
- 强正反馈来自真实履约：用户真的到场、活动顺利结束、且存在进一步社交结果
- 再约意愿是独立有效信号，需要单独写入记忆

### 6.6 找搭子能力：搜索优先 + 异步意向池

找搭子的首轮承接不再默认等价于“创建意向并入池”，而是先按用户偏好做一次**即时搜索**；只有当用户明确选择“继续帮我留意”时，才进入持续匹配。

#### 核心理念

**找搭子 ≠ 实时聊天室**

xu 当前的找搭子主链是：

## **即时搜索 + 明确下一步 + 可选异步意向池**

用户先表达“想找什么样的搭子”，系统先返回符合条件的候选结果；如果当下没有合适人选，或用户明确希望“xu 继续替我盯着”，才创建意向、进入意向池并通过消息中心 / 服务通知继续触达。

#### 核心场景

同一条 `find_partner` 主链必须统一承接这些表达：

- “想吃火锅，谁组我就去”
- “差一个麻将搭子”
- “泸州音乐节有人去吗”
- “周6平顶山有没有人”

它们本质上不是不同产品，而是同一条主链里的不同语义形态：

- 本地搭子
- 目的地同行
- 临时补位

#### 双阶段模型

##### 阶段 1：即时搜索（默认入口）

- 用户补充偏好后，系统先执行即时搜索
- 搜索结果以结果卡 / Swiper 形式展示
- Visitor 也可浏览结果
- 结果只展示必要摘要与匹配理由，不展示联系方式或隐私信息

此时用户可以选择：

- 看看这个人
- 登录后搭一下
- 登录后让 xu 帮我问问
- 换一批
- 继续帮我留意

##### 阶段 2：异步意向池（明确选择后进入）

- 只有当用户点击“继续帮我留意”时，才创建 `partner_intents`
- Logged-In / Unbound 用户先补手机号绑定，再进入持续匹配
- Registered 用户可直接入池
- 系统后台持续扫描并在命中后发送服务通知或消息中心任务卡

#### Pending Match 承接原则

- 匹配成功后，用户看到的不能只是一张“确认 / 取消”卡片
- 要先让用户理解“这次匹配是什么、谁该做什么、什么时候失效”
- 不新做一套匹配聊天室，而是在现有消息中心里展开详情态
- 消息中心卡片和详情态文案必须明确告诉用户“这是刚才那条找搭子任务的继续”

#### 表单原则

- 表单的职责是“描述想找什么样的人”，不是“提前把一场局定完”
- `textarea` 是搜索主信息源
- `费用方式`、`具体约点`、`谁订场` 不进入首轮搜索表单
- 内部标签体系仅用于搜索和排序，不对用户暴露为显式可选项

#### 搜索结果承接原则

- 搜索结果必须允许未登录用户先看
- Visitor 只能浏览结果，不能直接发起私聊或查看联系方式
- 结果区必须明确说明“现在还处于先搜一下阶段”
- 下一步动作要明确区分：
  - `登录后搭一下`
  - `登录后让 xu 帮我问问`
  - `登录后继续帮我留意`
  - `换一批`

## 7. 任务运行时

### 7.1 核心目标

核心目标不是“多一个状态机”，而是让系统第一次真的拥有同一件事。

### 7.2 任务对象至少要回答的 4 个问题

- 用户当前想完成什么事？
- 这件事现在推进到哪一步了？
- 下一步最合理的动作是什么？
- 最终有没有做成，结果是什么？

### 7.3 join_activity 阶段定义

| 阶段 | 含义 |
|------|------|
| `intent_captured` | 系统已识别“用户要找能参加的局” |
| `explore` | 已给出可参加活动列表，进入比较与选择 |
| `action_selected` | 用户已明确选择某个活动，准备报名 |
| `auth_gate` | 当前任务因登录/手机号绑定而挂起，等待恢复 |
| `joined` | 报名已成功写库 |
| `discussion` | 用户已进入活动讨论区，开始协作 |
| `post_activity` | 活动结束后进入复盘/再约跟进 |
| `done` | 已记录真实结果，任务完成 |

### 7.4 find_partner 阶段定义

| 阶段 | 含义 |
|------|------|
| `intent_captured` | 系统已识别“用户想找搭子” |
| `preference_collecting` | 正在收集活动类型、时间、地点等偏好 |
| `auth_gate` | 当前任务因登录/手机号绑定而挂起 |
| `intent_posted` | 搭子意向已成功写库 |
| `awaiting_match` | 系统正在后台等待下一次匹配 |
| `match_ready` | 已出现待确认匹配，等待用户或召集人决策 |
| `activity_created` | 匹配确认后已转成真实活动 |
| `done` | 已成局或已记录阶段性结果 |

### 7.5 create_activity 阶段定义

| 阶段 | 含义 |
|------|------|
| `intent_captured` | 系统已识别“用户想自己组一个局” |
| `draft_collecting` | 正在生成草稿，或用户正在修改地点/时间/人数 |
| `auth_gate` | 当前任务因登录/手机号绑定而挂起 |
| `draft_ready` | 草稿已真实写库，可继续调整或直接发布 |
| `published` | 活动已正式发布到活动域 |
| `done` | 发局任务已完成，进入后续分享/拉人阶段 |

### 7.6 产品意义

- 用户感知不再是“几个功能拼在一起”，而是“同一个 Agent 一直在帮我办”
- 我们第一次能做 outcome tracking，而不是只统计单点按钮点击
- 登录、通知、讨论区、活动后反馈，第一次被纳入同一条任务链

## 8. 关键产品模块

### 8.1 Visitor-First + Action-Gated Auth

- Visitor 可以浏览、探索、进入对话、查看详情
- 写入型动作统一先登录 + 绑定手机号
- auth gate 是任务挂起点，不是任务终止点

### 8.2 活动详情

- 活动详情页是 join / create / share / discussion 的重要承接面
- 活动详情不能只做静态信息页，必须承担报名、查看、联系、讨论预览等动作承接

#### 执行策略

活动详情采用**统一详情页（Page-First 执行）**。

这意味着：

- 对话入口负责发现与表达
- 报名、复制、克隆等确定性动作在详情页完成
- 详情页统一全屏承载，不再把 Chat Tool Mode 作为主流程分叉
- 保留半屏交互能力用于授权与局部补充（如 Auth Sheet），但不承担独立报名主链路
- 从首页、探索卡片、分享入口进入时，统一落到同一详情页与同一动作闸门

#### 在产品结构中的角色

- 对 `join_activity` 来说，详情页是从“看到这个局”走到“决定报名”的关键承接面
- 对 `create_activity` 来说，详情页是活动发布后的展示、分享与二次转化面
- 对 H5 来说，详情页本身也应尽可能承担“可分享的活动展示页”职责，而不必再单独依赖一套独立邀请函心智

#### 页面布局原则

- 使用统一全屏详情页，而不是碎片化弹层流
- 顶部导航明确、内容可完整滚动
- 活动核心信息、描述、参与者、讨论区预览应在同一页面中清楚展开
- 底部主 CTA 固定存在，优先承接“我要报名”

#### 报名动作原则

- 报名按钮始终清楚可见
- 点击报名后统一遵循 `Visitor-First + Action-Gated Auth`
- 如需登录或绑定手机号，应在详情页上下文内完成，不跳到割裂的外部流程
- 报名成功后统一进入讨论区，而不是停留在详情页只给一个 Toast

#### 讨论区预览

- 活动详情页应允许嵌入讨论区预览
- 讨论区预览的价值不是“多一块内容”，而是让用户在报名之前就感受到活动的真实氛围
- “查看更多”入口应把用户明确带向完整讨论区

#### 辅助动作

##### 复制文案

- 允许用户一键复制活动文案，方便转发到其他群或场景
- 复制文案的价值在于降低再分发成本，而不是替代正式分享能力

##### 我也组一个

- 允许用户基于当前活动快速克隆并发起类似活动
- 点击后回到首页并自动预填 AI 输入框
- 这个动作的目标是把 Joiner 平滑转成 Creator

**克隆逻辑**：

- 保留：活动类型、地点、人数、费用方式等主要结构
- 修改：时间自动调整为更合理的下一时段，如“明天”或“本周末”
- 转换：用户角色从 Joiner 转为 Creator

#### 成功指标

| 指标 | 说明 | 目标方向 |
|------|------|---------|
| 详情页到报名点击率 | 到达详情页后点击“我要报名”的比例 | 持续提升 |
| 报名链路完成率 | 点击报名后完成登录/绑定并报名成功的比例 | 持续提升 |
| 活动克隆使用率 | 点击“我也组一个”的比例 | 持续提升 |
| 文案复制使用率 | 点击“复制文案”的比例 | 持续提升 |

### 8.3 活动讨论区

- 活动讨论区不是独立社交产品，而是活动任务进入协作阶段后的承接面
- 报名成功后统一链路应进入讨论区
- 讨论区应帮助用户从“报上名”过渡到“真的协同起来”

#### 为什么需要活动讨论区

**核心场景**：

| 场景 | 说明 |
|------|------|
| 活动讨论 | 报名了火锅局后，讨论“几点集合”“谁先到” |
| 找搭子匹配后 | 匹配成功后，双方在这里沟通细节并转成真实活动协同 |
| 多人活动 | 多个参与者围绕同一活动同步信息 |

**产品定位**：

- 称为“活动讨论区”，而不是“群聊”或“聊天室”
- 强调它是围绕某个活动的讨论承接面
- 只有报名了活动的人才能参与
- 活动结束后讨论区自动归档

#### 作为任务阶段的定位

- 对 `join_activity` 而言，讨论区不是独立功能页，而是 `discussion` 阶段的主要承接面
- 用户从报名成功进入讨论区时，系统必须明确记录“任务已进入协作阶段”
- 后续活动提醒、活动后复盘 / 再约，都应从这条任务继续推进，而不是重新起一条无关对话

#### 核心功能

##### 实时消息

- 讨论区支持围绕活动的实时文字消息
- 用户发送消息后，消息应先经过内容安全检测，再持久化并广播

##### 在线状态

- 允许显示当前在线人数，增强协作感和临场感
- 在线状态用于帮助用户判断“现在是不是有人在看”

##### 历史消息

- 进入讨论区时自动加载最近消息
- 支持分页加载更多历史内容
- 历史消息应帮助后来加入的参与者快速补上下文

##### 离线通知

- 用户离线时，通过可用通知渠道提醒新消息
- 通知方式优先低摩擦方案，失败时再用兜底方案

#### 生命周期管理

| 活动状态 | 讨论区状态 | 说明 |
|---------|-----------|------|
| `active` | 开放 | 可发送消息 |
| `completed` | 归档 | 只读，不可发送 |
| `cancelled` | 归档 | 只读，不可发送 |

**归档行为**：

- 断开实时连接
- 保留历史消息可查看
- 禁止新消息发送

#### 内容安全

| 措施 | 说明 |
|------|------|
| 内容安全检测 | 检测敏感词和违规内容 |
| 敏感词过滤 | 违规内容拒绝发送，提示用户修改 |
| 用户举报 | 支持举报消息，管理员审核处理 |
| 违规记录 | 记录违规日志，用于风控分析 |

#### 成功指标

| 指标 | 说明 | 目标方向 |
|------|------|---------|
| 讨论区进入率 | 报名成功后进入讨论区的比例 | 持续提升 |
| 首条消息率 | 进入讨论区后发送第一条消息的比例 | 持续提升 |
| 成局率提升 | 有讨论区互动的活动相较无互动活动的成局率提升 | 持续提升 |

### 8.4 消息中心

消息中心不是传统意义上的“通知列表”，也不是第二个首页，而是：

## **任务收件箱 + 结果更新流**

它解决的是首页解决不了的那部分事：

- 首页负责判断“现在最该接住哪件事”
- 消息中心负责承接“那些不在此刻首屏，但仍然需要被继续的事”

#### 顶部层：等你处理

这是消息中心最重要的一层，优先展示需要用户现在做决定或继续推进的事项。

**典型内容**：

- 有个待确认匹配等你处理
- 你上次组的局还差 2 个人
- 你有个草稿还没发完
- 活动结束了，等你确认履约或补反馈
- 讨论区里有人在等你回应

**产品要求**：

- 每张卡都必须明确告诉用户：这是什么事、现在到哪一步、下一步该做什么
- 点击后恢复的必须是同一条任务，而不是跳到一个看起来无关的新页面
- 任务卡优先使用“继续处理”“查看进度”“现在确认”这类动作型文案，而不是模糊通知文案

#### 下层：结果更新

这一层才承接传统意义上的系统更新和异步变化，但仍然尽量挂回具体任务或活动。

**典型内容**：

- 有人报名了你的活动
- 有人加入了讨论区
- 匹配状态发生变化
- 活动开始前提醒
- 系统给出的阶段更新

**产品要求**：

- 每条更新尽量明确“它属于哪件事”
- 不做纯碎片化通知瀑布流
- 点击后优先进入对应任务、活动详情、讨论区或匹配详情态

#### 与首页的分工

- 首页只顶 1 个主状态
- 消息中心承接所有其他待处理事项和异步变化
- 首页负责“现在先处理什么”
- 消息中心负责“还有哪些事没丢”

#### 信息架构原则

- 顶部先展示 `任务收件箱`，按“等你处理 / 继续推进中”分组展示 active tasks
- 系统通知退居为 `结果更新`
- pending match、活动提醒、结果更新都应明确表现为“刚才那条任务的继续”
- 消息中心不是任务中心，但必须是任务恢复的重要入口

### 8.5 个人中心

- 个人中心负责用户资产、身份、偏好和已参与事项的归档
- 不承担首页主入口职责

### 8.6 H5 Web

H5 不再只是“小程序不可用时的降级方案”，而是承担大量真实产品实践的轻端承接面。

因此，H5 的定位是：

## **可独立承接完整核心链路的轻端产品，同时额外承担分享落点**

这意味着：

- H5 不再只是备用入口
- H5 需要能承接首页、对话、详情等核心能力
- 独立邀请函页面已经移除，分享能力由活动详情页承接
- 小程序仍然是微信内更强承接、更强通知和更低摩擦的版本，但不是唯一真身

#### H5 的主职责

- 承接首页与对话入口
- 承接活动详情浏览与后续动作
- 在小程序不可用、审核中、或用户处于非微信环境时，继续承担核心流程
- 作为跨平台传播落点，承接来自社交媒体、浏览器、线下物料等外部流量

#### 活动详情分享态的定位

活动详情页本身承担 H5 的外部分发职责。

更准确的说法是：

## **分享出去的活动详情，本身就是分享承接页。**

也就是说：

- 不再把“邀请函”继续当成一套独立产品心智
- 活动详情页需要足够清楚、足够适合分享
- 分享只是活动详情的一种访问来源，不是单独主流程

**产品判断**：

- 独立 `/invite/:id` 路由不再保留
- 外部分发、活动信息展示、报名与讨论入口统一由 `/activities/:id` 承接
- 不再维护一套独立邀请函页面心智

#### H5 结构

##### `/chat`

- `/chat` 就是 H5 当前的首页路由
- 它不是简单“降级聊天页”，而是 H5 的核心承接入口
- 它同时承担首页状态表达、对话承接、GenUI 返回和用户继续推进的主要动作
- 当小程序不可用时，H5 应尽可能承接完整核心功能

##### 活动详情分享态

- `/activities/:id` 是唯一活动详情与分享承接路由
- 分享出去的活动详情页本身承担外部分发职责
- 通过少量 UI 判断和展示变化，活动详情即可切换到更适合分享的展示态
- 独立邀请函能力已由活动详情吸收，不再继续维护一套独立页面

#### H5 与小程序的关系

- 两者共享同一套产品哲学，不允许长成两套不同世界观
- H5 更强调跨平台、轻承接、外部流量进入
- 小程序更强调微信内转化、通知触达、低摩擦完成动作
- 只要现实实践主要发生在 H5，就应优先保证 H5 的主链完整，而不是默认把完整体验都推迟到小程序

#### 执行原则

- H5 需要能承接完整核心链路，而不是只做展示页
- 不保留独立邀请函页面，分享入口统一落到活动详情
- 邀请函不是产品成立前提，也不是独立路由
- H5 不应独立长成另一套产品哲学
- `invite` 独立路由可移除，由活动详情分享态统一承接

### 8.7 页面结构与典型用户路径

#### 页面结构原则

- 首页和 chat 是同一个页面的两种状态，而不是两个并列页面
- H5 当前也遵守同一原则：`/chat` 就是首页路由
- 活动详情承担“查看 + 报名 + 分享展示 + 二次转化”的统一承接职责
- 消息中心不做复杂 dashboard，只通过 list item 的不同属性区分“任务”和“结果更新”

#### 核心页面清单

| 页面 | 形态 | 角色 |
|------|------|------|
| 首页 / Chat | 同一路由、同一页面的两种状态 | 首页状态承接 + 对话主舞台 |
| 活动详情 | 统一详情页 | 报名、查看、分享展示、讨论区预览、二次转化 |
| 活动讨论区 | 活动协作页 | 报名成功后的协作承接面 |
| 消息中心 | 任务收件箱 + 结果更新流 | 异步结果和任务恢复入口 |
| 个人中心 | 资产归档页 | 身份、偏好、活动资产管理 |
| 活动确认 / 草稿设置 | 任务中间承接页 | 发局确认与细节调整 |

#### 中间态组件清单

| 组件 | 作用 |
|------|------|
| `auth-sheet` | 登录 / 绑定手机号动作闸门 |
| `share-guide` | 发布成功后的分享引导 |
| `activity-preview-sheet` | 局部预览与轻量查看 |
| `thinking-bubble` | 对话处理中反馈 |
| `widget-skeleton` | 结构化卡片加载占位 |

#### 消息中心的列表区分方式

消息中心不单独做复杂分区页，而是在同一 list 中通过 item 属性区分不同类型：

- 需要立即处理的事项：更强标题、更明确动作文案、更高优先级样式
- 继续推进中的事项：展示当前阶段和下一步建议
- 结果更新：样式更轻，强调它属于哪件事

也就是说：

- 用 `title / subtitle / status / action / priority / badge` 这些 list item 属性完成区分
- 不引入额外复杂容器和信息架构

#### 典型用户路径

##### 发起者链路（Creator Flow）

首页空态 / 状态态  
→ 输入一句话或点击建议入口  
→ 进入同一页面内的对话主舞台  
→ 生成草稿 / 调整设置  
→ 发布成功  
→ 活动详情  
→ 分享出去（活动详情分享态）

##### 参与者链路（Joiner Flow）

首页 / 搜索 / 分享进来的活动详情  
→ 查看活动信息  
→ 点击报名  
→ 如需则走登录 / 绑定  
→ 报名成功  
→ 进入活动讨论区  
→ 后续由消息中心和活动后流程继续承接

##### 找搭子链路（Partner Flow）

首页输入“谁组我就去”或类似表达  
→ 对话中收集偏好  
→ 先看搜索结果  
→ 明确选择“搭一下 / 帮我问问 / 继续帮我留意”  
→ 如进入异步阶段，则由消息中心继续承接  
→ 命中匹配后进入详情态 / 后续协同链路

##### 回访链路（Resume Flow）

用户重新打开首页  
→ 首页只顶 1 个主状态  
→ 需要处理的其他事项进入消息中心  
→ 点击某条事项  
→ 恢复同一条任务，而不是重新开始

## 9. 合规性

### 9.1 合规原则

合规不是补丁，而是当前产品结构的一部分，尤其体现在讨论区、社交动作、通知和内容安全上。

### 9.2 当前包装策略

| 策略 | 说明 |
|------|------|
| 称为“活动讨论区” | 不称为“群聊”或“聊天室” |
| 强调“围绕活动的讨论” | 不是泛陌生人聊天 |
| 限制参与者 | 只有报名了活动的人才能参与讨论 |
| 自动归档 | 活动结束后讨论区自动归档 |

### 9.3 内容安全措施

| 措施 | 说明 |
|------|------|
| 内容安全检测 | 检测敏感词和违规内容 |
| 敏感词过滤 | 违规内容拒绝发送 |
| 用户举报机制 | 支持举报，管理员审核处理 |
| 违规记录 | 记录违规日志，用于风控分析 |

### 9.4 审核准备

- 需要准备清楚的功能说明，说明讨论区是围绕活动的讨论功能
- 需要准备内容安全说明
- 需要为审核与平台变更留出时间缓冲

## 10. 记忆与个性化

### 10.1 Working Memory

Working Memory 不是为了“显得更聪明”，而是为了让 xu 真正具备“续上同一件事”的能力。

### 10.2 它要记住什么

| 维度 | 示例 |
|------|------|
| 活动类型偏好 | 喜欢火锅、不喜欢 KTV |
| 时间偏好 | 偏好周末、晚上 |
| 地点偏好 | 常去观音桥、解放碑 |
| 社交偏好 | 喜欢小规模、怕太吵、不爱大群 |
| 真实结果 | 到没到场、顺不顺利、有没有再约 |

### 10.3 它为什么重要

- 让首页和消息中心不只是“记住任务”，还能记住这个人平时怎么做选择
- 让找局、找搭子、活动后续上不再每次都从零开始
- 让 xu 更像一个会张罗、记性好、知道你习惯的人，而不是每次都重新问一遍的助手

### 10.4 记忆原则

- 轻信号可以记，但不能被误当成强偏好
- 强信号来自真实结果，例如真实到场、活动顺利结束、明确再约意愿
- 记忆的目的不是“多记”，而是让下一步更像在继续同一件事

### 10.5 使用方式

- 偏好与历史结果可被写入长期记忆
- 后续推荐、欢迎态、活动后续上、再约建议，都应优先参考这些真实信号
- 记忆应服务主流程，而不是长成一套单独的画像系统

## 11. 视觉与语气

### 11.1 视觉总原则

- Welcome 可以有进入感和情绪，但不能喧宾夺主
- Conversation 必须收敛，让文本与 GenUI 卡片成为主角
- 高级感优先通过排版、间距、字号、黑白灰层级完成，不靠大面积蓝紫渐变或玻璃态

### 11.2 UI 主风格

产品既然采用生成式 UI，整体风格应坚持：

#### 不是内容流

因为你不是靠 feed。

#### 不是表单堆叠

因为你不是纯后台工具。

#### 不是空白聊天框

因为你不是通用助手。

#### 而是

## **轻首页 + 任务卡 + 对话流 + 动态结果卡**

用户感受到的应该是：

- 这个产品懂我现在在干嘛
- 它不是要我从零想 prompt
- 它会自己把下一步长出来

### 11.3 语气规范

- 不装，不端，不说“全息构建”“意图向量”这种词
- 要像一个会张罗的人，而不是一个炫技助手
- 语气目标是：接地气、能办事、让人愿意继续往下说

#### AI 角色设定

| 属性 | 设定 |
|------|------|
| 名字 | `xu` |
| 定位 | 碎片化社交助理，不是平台入口 |
| 称呼用户 | 优先直接对话，不刻意套近乎 |
| 自我介绍 | “今天有什么想玩的？” |
| 语气 | 短句、直接、会张罗，但不过度热情、不装熟 |

**原则**：

- xu 是一个助理，而不是一个平台入口
- 活动、搭子、讨论区、复盘和推荐都应表现为 xu 的能力模块，而不是互相割裂的页面
- 不要让 UI 的高级感变成“距离感”

#### xu 回复示例

| 不要这样说 | 应该这样说 |
|-------------|-----------|
| “已为您构建全息活动契约，请确认地理围栏坐标。” | “帮你把局组好了！就在观音桥，离地铁口 200 米，把这张卡片发群里吧。” |
| “正在解析您的意图向量...” | “收到，我先帮你整理一下。” |
| “活动实体已成功持久化至数据库。” | “卡片做好了！看看这样安排行不行？” |
| “检测到地理位置偏移，建议重新校准。” | “这个地址好像有点远，要不要换一个？” |

#### 按钮文案

| 不要这样写 | 应该这样写 |
|------------|-----------|
| 确认提交 | 确认发布 |
| 执行分享 | 分享到群 |
| 调整地理坐标 | 调整位置 |
| 查看详细信息 | 查看详情 |
| 展开地图视图 | 展开地图 |

#### 错误提示

| 不要这样写 | 应该这样写 |
|------------|-----------|
| “解析失败，请检查输入格式。” | “抱歉，我没理解你的意思，试试换个说法？” |
| “网络请求超时，错误码 504。” | “网络有点慢，再试一次？” |
| “今日配额已耗尽。” | “今天的创建活动额度用完了，明天再来吧～” |
| “用户认证失败。” | “需要先绑定手机号才能继续哦” |

#### 空状态文案

| 场景 | 建议文案 |
|------|---------|
| 无待参加活动 | “最近没局？有些朋友可能在等你开口。” |
| 探索无结果 | “附近暂时没有活动，要不你来发起第一个？” |
| 消息中心为空 | “还没有需要处理的事，有新进展我会帮你记着。” |
| 输入框占位符 | “粘贴群聊天记录，或直接告诉我怎么玩...” |

#### AI 思考与反馈文案

| 场景 | 建议文案 |
|------|---------|
| AI 正在生成 | “收到，我先帮你整理一下。” |
| 系统生成完毕 | “卡片做好了！看看这样安排行不行？” |
| 发布成功 | “搞定！活动已发布，快分享给朋友吧。” |
| 额度用完 | “今天的创建活动额度用完了，明天再来吧～” |
| 网络错误 | “网络有点慢，再试一次？” |
| 解析失败 | “抱歉，我没理解你的意思，试试换个说法？” |

## 12. 结果观察（非主组织维度）

### 12.1 组织原则

本 PRD 不围绕“成功指标”组织，也不把产品是否成立建立在短期数字之上。

更重要的是：

- 守住精神内核
- 守住产品本质
- 守住主流程
- 持续把真实场景打磨顺

### 12.2 弱化指标的原因

- 现在还不是靠指标驱动收口的时候
- 太早把注意力放在数字上，容易把产品写成“怎么证明”，而不是“我们到底在做什么”
- 对产品早期来说，方向是否清楚、流程是否顺、表达是否真诚，比短期转化数字更重要

### 12.3 可观察的结果信号

当主流程稳定之后，再逐步观察这些结果型信号：

- 用户是否真的能完成 `join_activity`
- 用户是否真的能完成 `create_activity`
- 用户是否真的愿意进入 `find_partner`
- 被 auth gate 挂起的任务能不能恢复
- 报名后的讨论区和活动后续上有没有被真正接住

这些信号属于后续观察项，不是本文档的主组织维度。

## 13. 实现优先级

这部分用于表达产品落地顺序。

### 13.1 推荐推进顺序

1. 先做首页 3 态
- 空态
- 进行中
- 待出发

2. 再做 3 条核心任务链路
- 找局
- 发起组局
- 补人

3. 再做一个活动详情页
- 让用户能真正报名、查看、联系

4. 最后补
- 待继续
- 收尾态
- 次要事项区
- 更多记忆动态化

### 13.2 结构结论

产品定义不再停留在“首页长什么样”，而是明确为以下结构：

**群主分身型 agent 产品**  
**首页是状态机，不是活动广场**  
**生成式 UI 负责任务过程，而不是替代产品结构**

- 历史废弃方案不再占据主干章节；如确有保留必要，可沉到附录或单独决策记录

## 附录二：术语表

| 术语 | 说明 |
|------|------|
| `xu` | 产品内的助理角色名，也是统一用户感知入口 |
| `群主分身` | 产品核心比喻：不是平台，不是空白助手，而是替人把局张罗起来的数字分身 |
| `组局操作系统` | 指围绕找局、发局、补人、跟进结果这套工作流的持续承接系统 |
| `状态首页` | 首页不是广场，也不是纯 chat，而是优先判断“现在最需要被接住的事”的状态容器 |
| `agent 流` | 理解需求、调用数据、推进流程的核心执行层 |
| `GenUI` | 生成式 UI 承接层，负责把结果、进度、下一步动作长出来 |
| `GenUI blocks` | 当前统一结构化块协议，核心包括 `text`、`choice`、`entity-card`、`list`、`form`、`cta-group`、`alert` |
| `Widget_*` | 旧实现与兼容层命名，可继续存在，但不再作为 PRD 的主叙事 |
| `join_activity` | “帮用户找到并加入一个局”的连续任务主链 |
| `create_activity` | “帮用户把一个局发出去”的连续任务主链 |
| `find_partner` | “帮用户找搭子 / 同行 / 补位”的连续任务主链 |
| `Agent Task Runtime` | 持续拥有同一件事的任务运行时中枢对象 |
| `H0 / H1 / H2 / H3 / H4` | 首页状态机的 5 种首页表达态：空态、待继续、进行中、待出发/待处理、收尾态 |
| `Visitor-First` | 默认允许用户先浏览、先体验、先开口，再逐步进入动作闸门 |
| `Action-Gated Auth` | 报名、发布、联系方式等高价值动作统一要求登录 + 手机号绑定 |
| `活动详情分享态` | 不再单独强调“邀请函”页面，分享出去的活动详情本身就承担邀请函职责 |
| `活动讨论区` | 围绕某个活动的讨论承接面，不是泛陌生人聊天室 |
| `任务收件箱` | 消息中心顶部层，优先承接需要用户立即处理或继续推进的事项 |
| `结果更新流` | 消息中心下层，承接异步变化与阶段更新，但仍尽量挂回具体任务或活动 |
| `Working Memory` | 为了让 xu 真正“续上同一件事”的偏好与真实结果记忆层 |
| `轻信号` | 例如报名、浏览、短时表达等，不能直接等价为强偏好或强正反馈 |
| `强信号` | 例如真实到场、活动顺利结束、明确再约意愿等，可作为更可靠的判断依据 |
| `即时搜索 + 可选异步意向池` | 当前找搭子主模型：先搜，再决定是否进入持续匹配 |
| `Pending Match` | 已出现待确认匹配，等待用户或临时召集人推进的状态 |
| `Temp_Organizer` | 匹配成功后负责确认并转为真实活动的临时召集人 |
| `不做平台，做工具` | xu 的差异化定位原则，不走活动货架/电商平台路线 |
