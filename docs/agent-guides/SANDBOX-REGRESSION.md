# 多账号沙盘回归清单

## 目标

用固定 5 个测试账号，反复模拟多账号交互场景，确认核心业务链路、通知链路、AI 会话权限是否正常。

## 执行命令

```bash
bun run sandbox:regression
```

只跑某一个场景：

```bash
bun run sandbox:regression --scenario capacity
```

## 当前场景

### 1. `basic-discussion-flow`
- 1 人创建活动
- 4 人报名
- 3 人发送讨论区消息
- 校验公开详情、群聊消息、群聊列表

### 2. `capacity-limit`
- 活动容量设为 4（含发起人）
- 前 3 个报名成功
- 第 4 个报名用户被拦截
- 校验“活动人数已满”

### 3. `duplicate-and-rejoin`
- 同一用户重复报名被拦截
- 退出活动后可以重新加入
- 校验人数正确回滚和恢复

### 4. `permission-guards`
- 创建者不能报名自己创建的活动
- 未报名用户不能发讨论区消息
- 非创建者不能修改活动状态

### 5. `cancel-visibility`
- 活动取消前可公开访问
- 活动取消后公开详情不可访问
- 取消后报名被拦截

### 6. `notifications-flow`
- 报名后创建者收到 `join` 通知
- 标记已读后未读数回落
- 取消活动后报名用户收到 `cancelled` 通知
- 消息中心能统计通知未读

### 7. `ai-access-flow`
- 游客可获取欢迎卡并发起 AI 聊天
- 登录用户 AI 会话会持久化到 `/ai/conversations`
- 其他账号不能查看别人的 AI 会话
- 其他账号不能劫持别人的 `conversationId`

## 前置条件

- 已执行 `bun install`
- 数据库可用
- 根目录 `.env` 已配置
- 当前环境不是 `production`
- Admin 超级验证码可用
- AI 模型相关环境变量可用（用于 `ai-access-flow`）

## 输出解读

- 每个场景会输出 `PASS` / `FAIL`
- 最后会输出 `Sandbox Summary`
- 任意一个场景失败，脚本会返回非 0 退出码

## 建议使用方式

- 每次改报名、活动状态、群聊、公开详情、通知、AI 会话逻辑后都跑一遍
- 发布前至少跑完整套一次
- 如果只改局部功能，先用 `--scenario` 跑对应场景
